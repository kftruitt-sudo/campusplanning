{%- comment -%} Story layout — the standalone, full-document reading experience for a single Telar story. Unlike most pages it does NOT inherit the default shell; it is its own complete HTML document so it can own the scroll surface, fixed navigation, and viewer plates without the site header/footer competing for the viewport. What it renders — the document (same cascade-layer + Bootstrap + theme-font setup as the default layout, plus a block of inline JS that hands localized strings and config flags to the client), then a containing: a fixed back navigation button, a floating share button, the scroll surface, the intro card (title, subtitle, optional byline/description, optional table of contents, and any author-facing configuration warnings), and a hidden `step-data` block. That hidden block is the heart of the page — for each story step it emits a `story-step.html` include carrying the question, answer, object reference, and viewport coordinates (x/y/zoom/page) plus clip and layer metadata; card-pool.js reads these at init to build the visual scroll-driven cards. The layout closes by loading OpenSeadragon, the Telar core and story bundles, and conditionally the unlock overlay for protected stories. Key context — `page.data_file` names the story's data file under `site.data`; its first entry may be a `_metadata` row carrying viewer warnings and feature flags (e.g. has_latex). `lang` resolves from `site.telar_language` with an English fallback and supplies every user-facing string. Theme fonts derive from `site.data.themes[site.telar_theme]`. Non-obvious behaviour — encrypted (protected) stories take a deliberately leak-free path: jekyll-seo-tag is bypassed in favour of a generic description and canonical URL, the byline and description are suppressed from the intro card, and the steps are NOT rendered server-side (an empty container is left for story-unlock.js to populate after decryption), so authored content never ships in the HTML before the reader supplies the key. KaTeX is loaded lazily by assets/js/katex-loader.js, only after storyData (or, for protected stories, has_latex frontmatter) reveals the story uses LaTeX.
